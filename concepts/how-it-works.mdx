title: How it works
description: A detailed walkthrough of TSS's zero-config runtime CSS generation lifecycle — from script load to live DOM observation.
---

<Note>
  TSS does **not** require a build step. Drop in the script tag and it works immediately with any server-rendered or static HTML page.
</Note>

TSS generates utility CSS at runtime by scanning the DOM, parsing class names, and injecting `<style>` elements directly into `<head>`. There is no compilation, no purging, and no configuration file needed.

## Lifecycle

<Steps>
  <Step title="Script auto-init">
    When the TSS script is evaluated, it checks `document.readyState` before doing anything else.

    ```js
    if (typeof document !== 'undefined') {
      restoreThemePreference();
      if (document.readyState === 'loading') {
        document.addEventListener('DOMContentLoaded', init);
      } else {
        init();
      }
    }
    ```

    - If the document is still parsing (`'loading'`), TSS binds `init` to the `DOMContentLoaded` event so the full DOM is available before scanning begins.
    - If the document has already parsed (e.g., the script tag is at the bottom of `<body>`, or deferred), `init()` is called immediately.
    - `restoreThemePreference()` runs **before** `init()` in both cases, reading any saved theme from `localStorage` and applying `.dark` or `.light` to `<html>` before the first paint.
  </Step>

  <Step title="Theme injection">
    `init()` creates a `<style id="tss-theme">` element and sets its content to the return value of `buildThemeCSS()`. This single call injects:

    - A universal box-sizing reset
    - A system font stack on `html`
    - All semantic color tokens as CSS custom properties on `:root` (light-mode values)
    - A `@media (prefers-color-scheme: dark)` block that overrides those same custom properties with dark-mode values
    - `html.dark { ... }` and `html.light { ... }` class overrides for manual control

    ```js
    _themeStyle = document.createElement('style');
    _themeStyle.id = 'tss-theme';
    _themeStyle.textContent = buildThemeCSS();
    document.head.appendChild(_themeStyle);
    ```
  </Step>

  <Step title="Utilities style tag">
    A second, initially empty `<style id="tss-utilities">` element is appended to `<head>`. All utility CSS rules generated from class names are written into this element — keeping theme tokens and utility rules cleanly separated.

    ```js
    _utilStyle = document.createElement('style');
    _utilStyle.id = 'tss-utilities';
    document.head.appendChild(_utilStyle);
    ```
  </Step>

  <Step title="DOM scan">
    `scanNode(document.body)` walks the entire document body. It processes the root element's own class list first, then uses `querySelectorAll('*')` to iterate every descendant element and call `processClassList()` on each one.

    ```js
    function scanNode(root) {
      if (!root || !root.querySelectorAll) return;

      if (root.classList) {
        processClassList(root.classList);
      }

      const elements = root.querySelectorAll('*');
      for (let i = 0; i < elements.length; i++) {
        processClassList(elements[i].classList);
      }
    }
    ```
  </Step>

  <Step title="Class parsing">
    Each class name is passed through `parseClass(className)` in `parser.js`. The function tries a chain of specialized parsers in order — spacing, color, layout, typography, borders, flex, grid, and more.

    - If a parser matches, it returns a `{ selector, rules }` object where `selector` is the escaped CSS selector and `rules` is a plain object of CSS property-value pairs.
    - If no parser matches, `parseClass` returns `null` and the class name is **silently ignored** — unrecognized classes never cause errors.
  </Step>

  <Step title="Rule generation and batched flush">
    `generateRule(className)` is called for each class name. It is the central function that ties parsing and output together:

    ```js
    function generateRule(className) {
      if (_cache.has(className)) return;
      _cache.add(className);

      const result = parseClass(className);
      if (!result) return;

      const { selector, rules } = result;

      let css = `${selector} {\n`;
      for (const [prop, val] of Object.entries(rules)) {
        css += `  ${prop}: ${val};\n`;
      }
      css += '}\n';

      _pendingCSS += css;

      if (!_rafScheduled) {
        _rafScheduled = true;
        requestAnimationFrame(flushPending);
      }
    }
    ```

    Generated CSS strings accumulate in `_pendingCSS`. Rather than writing to the DOM on every class, a single `requestAnimationFrame` callback (`flushPending`) appends the entire batch to `_utilStyle.textContent` at once, avoiding repeated style recalculations.

    ```js
    function flushPending() {
      if (_pendingCSS) {
        _utilStyle.textContent += _pendingCSS;
        _pendingCSS = '';
      }
      _rafScheduled = false;
    }
    ```
  </Step>

  <Step title="MutationObserver">
    After the initial scan, `startObserver()` attaches a `MutationObserver` to `document.body` that keeps TSS in sync with dynamic content:

    ```js
    observer.observe(document.body, {
      childList: true,
      subtree: true,
      attributes: true,
      attributeFilter: ['class'],
    });
    ```

    - **`childList` mutations** — when new elements are inserted into the DOM, each added node of type `Element` is passed to `scanNode()` so their classes are processed immediately.
    - **`attributes` mutations on `class`** — when JavaScript adds or removes classes from an existing element (e.g., toggling state), `processClassList()` is called on that element's updated class list.

    This means TSS works correctly with client-side frameworks, JS-driven UI updates, and any runtime class manipulations without any extra configuration.
  </Step>
</Steps>

## The `_cache` Set

TSS maintains a module-level `Set` called `_cache` to ensure every class name is processed **at most once**, regardless of how many elements use it or how many times it appears.

```js
const _cache = new Set();
```

At the top of `generateRule`, before any parsing occurs:

```js
if (_cache.has(className)) return;
_cache.add(className);
```

If the class has already been processed, the function returns immediately. If not, it's added to the cache before parsing. This prevents duplicate CSS rules from accumulating in `_utilStyle` as new elements are added or class attributes change.

## Class name escaping

CSS selectors cannot contain unescaped special characters like `/`, `[`, `]`, `:`, or `%`. TSS handles this automatically using the `escapeSelector()` helper in `parser.js`:

```js
function escapeSelector(str) {
  return str.replace(/([.:#\[\]()>+~=!@$%^&*,{}'"/\\])/g, '\\$1');
}
```

This means classes like `w-1/2`, `grid-cols-3`, or any class containing bracket notation are safely escaped before being written into the stylesheet. For example, the class `w-1/2` produces the selector `.w-1\/2` in the generated CSS, which the browser correctly matches back to the element.
